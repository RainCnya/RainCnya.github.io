1001 布布吃地瓜

题意：

给定一个网格，网格中有数字，从左上角走到右下角，每一条路径的贡献值为当前路径中不存在的最小非负整数。找出最小的贡献值作为答案。

思路：

如果一条路径上没有 0，那么这条路线的贡献值就是 0，即为最小。因此，可以直接从 0 枚举到 n∗mn*mn∗m，超过 n∗mn*mn∗m 的值不用统计，证明略。

连通性对偶结论：在四连通的网格中，从 (1,1)(1, 1)(1,1) 到 (n,m)(n, m)(n,m) 的路径被阻断，当且仅当阻断点（数值为 kkk 的点）通过八连通的方式，形成了一条连接 “左/下边界” 和 "右/上边界" 的屏障。

因此我们可以定义两个边界集合：AAA 表示右/上，BBB 表示左/下。如果有一组数值为 kkk 的连通块同时触碰到了 AAA 和 BBB，那么所有从起点到终点的路径都必然会经过 kkk，此时 MEX{S}≠kMEX\{S\} \neq kMEX{S}=k。

考虑到这是一个连通性问题，我们可以采用并查集这个数据结构实现。接着取出所有数值为 kkk 的点，然后合并merge，再判断是否阻断路线即可。（ 注意特判起点和终点的数值 ）

复杂度分析，理论上每个点只访问了一次，因此为 O(mn⋅α(mn))O(mn \cdot \alpha(mn))O(mn⋅α(mn))， 后面那一项是并查集复杂度，通常为 O(1)O(1)O(1)。

1004 左右脑互搏

题意：

给定 n 个数，每回合可以删除一个数，A 和 B 轮流操作，操作不了的人输了。
限制条件：删除的这个数 必须大于 删除这个数后集合的异或和。

思路：

首先定性，这是一道博弈论的题目，而且是一个典型的公平组合游戏（这个词不重要），引入一个博弈论的概念，必胜状态 (P) 和必败状态 (N)。

  • 必胜态：从当前的状态出发，所有能到达的后续状态都是必胜态。
  • 必败态：从当前的状态出发，存在一条路径到达下一个必败态。

注：这里的必胜必败只与当前状态相关，而与玩家无关！

由于这个限制条件比较麻烦，所以不能用 Nim 游戏的 SG函数 来计算，接着注意到 n≤20n \leq 20n≤20，这启发我使用 O(2n)O(2^n)O(2n) 复杂度，仔细想想，有两个算法待选：状态压缩，DFS。再动一动脑，就选择记忆化搜索了。

定义 Memo[mask]Memo[mask]Memo[mask] 为当前状态的胜负情况：-1 尚未计算，0 必败态（Right赢），1 必胜态（Left赢）。

这个 Memo 是 memory 的缩写，一般用于记忆化搜索（DP的另一种实现方式），mask 位掩码 ( 二进制数 ) ，第 iii 位是 1 表示第 iii 位集合中，反之不在。

定义完状态，考虑转移：对于当前的 mask，我们需要遍历每一个二进制位 1 的位置 iii，然后判断删完之后是否存在一条路径到达必败态，如果没有，就说明当前是必胜态。

接着考虑优化异或运算，作为位运算，它具有一些良好的性质，比如 “自反性”：A⊕B⊕A=BA \oplus B \oplus A = BA⊕B⊕A=B，所以只需要记录当前状态的 xorSum 异或和，然后下一级状态的异或和就可以简单的写为：xorSum ^ a[i]。从而优化了遍历集合的复杂度。

题意：

状态转移：参考 0/1 背包，对于每一个加成，记录其对选中三项属性的增量 (g1,g2,g3)(g_1, g_2, g_3)(g1​,g2​,g3​) 和总增量 sgsgsg。

dp[v1][v2][v3]=min⁡(dp[v1][v2][v3], dp[v1−g1][v2−g2][v3−g3]+sg)dp[v_1][v_2][v_3] = \min(dp[v_1][v_2][v_3], \ dp[v_1-g_1][v_2-g_2][v_3-g_3] + sg) dp[v1​][v2​][v3​]=min(dp[v1​][v2​][v3​], dp[v1​−g1​][v2​−g2​][v3​−g3​]+sg)

注：为了符合 0/1 背包的要求，需采用逆序遍历。

最后统计答案，满足 dp[v1][v2][v3]≤150dp[v_{1}][v_{2}][v_{3}] \leq 150dp[v1​][v2​][v3​]≤150，就根据答案算出当前属性值下的伤害总和，然后更新全局最大值即可。

你以为这就结束了吗？这题还卡常数，虽然我们的复杂度没问题，但是还是会TLE，所以再次考虑优化，发现即便所有的加成总增量加起来也不会超过 150，这意味着我们根本就不需要在 DP 中记录最小的消耗点数，因为即便全选也不会超标。

此时，问题就从 “求最小消耗” 转化为 “判定是否可达”，既然如此，开一个 bool 数组就能满足了，不妨再拓展一步，都用 bool 了，为什么不用 bitset 再优化呢？于是就这么水过去了。
