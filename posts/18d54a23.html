3751：快速排序

一、 题目描述

  • 给定 n 个整数，请对他们进行排序。
  • 输入：多组数据， n ≤ 100000，每组数据有 n 个。
  • 输出：每行输出排序后的 n 个整数。

特别说明：

对于快速排序来说，只需要大概掌握思维方式即可，无需记忆具体代码实现。

一般来说我们会使用 std::sort() 函数来实现快速排序。

二、 思路分析

1. 核心思路

这题思路很简单：只需要对数据进行排序即可。

2. 最容易想到的暴力算法：冒泡排序

当我们遇到 排序 的时候，最容易想到的一种解法就是 冒泡：

冒泡排序：具体实现可以看我的另一个笔记博客。

3. 为什么冒泡不行

冒泡排序虽然简单，但它能通过这道题吗？我们来看题目的数据范围：N ≤ 100, 000。

  • 复杂度分析： 冒泡排序有两层嵌套循环。
    • 外层循环 N − 1 次。内层循环平均约 N/2 次。
    • 总的时间复杂度是 O(N2)。
  • 计算量估算：
    • 当 N = 1e5 时， N2 = (1e5)2 = 1e10。
    • 评测机 1 秒钟大概能执行 1e8 次计算。
    • 1e10 次计算大约需要 100 秒！
    • 而题目的时间限制是 3 秒。

现在能理解为什么冒泡排序会 TLE 了吧。

结论： O(N2) 的算法面对 N = 105 的数据规模，会严重超时 (TLE)。

我们必须寻找更高效的算法。

4. 优化进行中… O(Nlog N)

从复杂度的角度思考，O(n2) 的算法不行，那就往下试试看 O(nlog n) 的算法。

而这类算法的核心一般是 分治 （Divide and Conquer）。

▶
什么是分治？

一般来说，分治就是把一个大问题，拆解成若干个规模更小，性质相同的子问题。

通过递归查询子问题的解，最后合并成整体的答案。

举个例子，你要考察 40 个人的打卡情况。可以分配 4 个小组长，让他们每个人负责 10 个人，然后你只需要负责接受这 4 个小组长的回复即可。

5. 其实也不难的优化方案：快速排序

快速排序的策略也是如此。

核心思想：

  1. 分：从数组中挑选出一个 基准 p （Pivot），然后对于所有小于 P 的元素都放到它左边，所有大于 P 的元素都放到它右边。

  2. 治：递归地对基准左边和右边的两个子区间进行快速排序。

  3. 合：无需操作。因为“分区”步骤已经保证了左区间的数都小于右区间的数，当左右两半各自有序后，整个数组就自动有序了。

5.1. 分区的具体实现：

（此处采用双指针法）：

  1. 选一个基准，例如 p = a[(l + r)/2]。
  2. 设置指针 i = l, j = r。
  3. i 向右移动，直到 a[i] >  = p。
  4. j 向左移动，直到 a[j] <  = p。
  5. 若 i < j，交换 a[i] 和 a[j]，然后 i + +, j − −。
  6. 重复 3-5 步，直到 i > j。

在一次 分区 后，数组被分为 [l...j] 和 [i...r] 两部分。

5.2. 分治子区间

递归地对左子区间 [l, j] 进行快速排序。

递归地对右子区间 [i, r] 进行快速排序。

5.3. 合并子区间

无需操作。

三、 关键点与易错点

  • 递归边界：if (l >= r) return;。

    • (当区间为空或只有1个元素时，必须停止)。

  • 复杂度：快速排序的 平均 时间复杂度是 O(Nlog N)。

  • 最坏情况：如果每次 基准 都是最大或最小值（例如在一个已排序的数组上总是选第一个元素），那么 分区 会极不均匀（0 和 N − 1），导致递归树退化成 O(N) 层，总复杂度退化为 O(N2)。

  • 稳定性：如上文所述，如果存在最坏情况，那么算法的复杂度会退化，这种情况就说明这个算法不稳定。

  • 规避最坏情况：

    • 可以采用下面几种方案：
    • “随机”，“三点取中”，“直接取中点”。

但总会存在不做人的出题人出阴间数据卡时间……

四、 复杂度分析

算法 平均时间复杂度 最坏时间复杂度 空间复杂度
冒泡排序 O(n2) O(n2) O(1)
快速排序 O(nlog n) O(n2) O(log n)

梳理如下：

  • 快速排序 的平均时间 O(Nlog N) 远优于冒泡排序的 O(N2)，能够轻松通过 N = 105 的数据。

  • 它的空间复杂度 O(log N) 来自于递归调用栈的深度，也远优于归并排序的 O(N) 辅助数组。
